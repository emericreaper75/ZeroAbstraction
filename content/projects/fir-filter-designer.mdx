<ExecutiveSummary>
The FIR Filter Designer is a browser-based tool for designing finite impulse response digital filters. It runs entirely client-side — no server required — with the computationally intensive Parks-McClellan (Remez exchange) algorithm compiled from Rust to WebAssembly for near-native performance.

This project democratises access to optimal filter design by removing the need for a MATLAB or Python installation, providing engineers with a 60 fps interactive design loop directly in the browser.
</ExecutiveSummary>

<ProblemContext title="The Problem">
Existing FIR design tools require either a MATLAB/Python installation (barrier to entry) or use suboptimal window methods (Kaiser, Hamming) that produce filters with 10–15% more coefficients than necessary for the same specification.

By putting the equiripple-optimal Parks-McClellan algorithm in the browser with real-time D3.js frequency response rendering, engineers can iterate on filter specifications interactively. 

> **Note.** The Parks-McClellan algorithm guarantees the minimum filter order for a given set of passband ripple, stopband attenuation, and transition bandwidth specifications. For a typical lowpass design with $0.1\,\text{dB}$ passband ripple and $60\,\text{dB}$ stopband attenuation, Parks-McClellan requires order $N = 63$ vs. $N = 71$ for the Kaiser window method — an $11\%$ reduction in computational cost per filtered sample.
</ProblemContext>

## Parks-McClellan Background

### The Minimax Problem

Given a desired frequency response $D(\omega)$, a weighting function $W(\omega)$, and a set of constrained frequency bands $F$ (passbands and stopbands, excluding transition bands), find the Type I linear-phase FIR filter $H(\omega)$ of order $N$ that solves:

$$
\min_{h[n]} \; \max_{\omega \in F} \left| W(\omega) \left[ D(\omega) - H(\omega) \right] \right|.
$$

For a symmetric FIR filter ($h[n] = h[N-n]$), the frequency response has the form:

$$
H(\omega) = e^{-j\omega N/2} \sum_{k=0}^{M} a[k] \cos(k\omega), \quad M = \lfloor N/2 \rfloor,
$$

reducing the problem to finding $M+1$ real coefficients $\{a[k]\}$.

### The Chebyshev Alternation Theorem

The optimal solution exists, is unique, and satisfies the **alternation theorem**: the weighted error achieves its maximum magnitude at $\geq M+2$ distinct frequencies $\omega_1 < \omega_2 < \cdots$ with alternating sign:

$$
W(\omega_i)\left[D(\omega_i) - H(\omega_i)\right] = (-1)^i \delta, \quad i = 1, \ldots, M+2,
$$

where $\delta$ is the equiripple error level. This characterisation is both necessary and sufficient for optimality, meaning any filter satisfying the alternation condition is guaranteed to be the unique minimax-optimal solution.

<ImplementationJourney>
  <JourneyMilestone date="Q1 2024" title="Initial Prototype in TypeScript">
    The first version was written entirely in TypeScript to validate the math. It worked, but performance fell apart for filter orders $>128$. The UI thread would freeze for $200\text{ms}$ while computing a 256-tap filter, ruining the interactive experience.
  </JourneyMilestone>
  <JourneyMilestone date="Q2 2024" title="Rust + WASM Migration">
    Ported the core Remez exchange loop to Rust. Used `wasm-bindgen` to expose the function. We observed a $15\times$ speedup, but memory leaks occurred due to improper handling of `Float64Array` memory boundaries.
  </JourneyMilestone>
  <JourneyMilestone date="Q3 2024" title="Memory Optimisation & D3 Canvas">
    Fixed the memory issues by keeping a single allocated buffer in WASM memory and passing pointers, rather than serializing/deserializing arrays. Moved plotting from Recharts to a custom D3 Canvas implementation to unblock the main thread.
  </JourneyMilestone>
</ImplementationJourney>

## WebAssembly Core

The Remez exchange algorithm is implemented in Rust and compiled to WebAssembly via `wasm-pack`. The core function:

```rust
use wasm_bindgen::prelude::*;
use std::f64::consts::PI;

#[wasm_bindgen]
pub fn design_fir(
    order: usize,
    bands: &[f64],      // [f1_start, f1_end, f2_start, f2_end, ...]
    desired: &[f64],     // [d1, d2, ...] one per band
    weights: &[f64],     // [w1, w2, ...] one per band
    max_iter: usize,
) -> Vec<f64> {
    let n_coeffs = order + 1;
    let m = order / 2;  // Type I assumption (even order)
    let n_extremals = m + 2;
    
    // Initialise extremal frequencies uniformly across bands
    let mut extremals = init_extremals(bands, n_extremals);
    let mut coeffs = vec![0.0; n_coeffs];
    
    for _iter in 0..max_iter {
        // Step 1: Build and solve the interpolation system
        let (a, delta) = solve_alternation_system(
            &extremals, m, bands, desired, weights
        );
        
        // Step 2: Evaluate weighted error on dense frequency grid
        let n_grid = 2048;
        let (grid, errors) = evaluate_error_on_grid(
            &a, n_grid, bands, desired, weights
        );
        
        // Step 3: Find new extremal set via exchange
        let new_extremals = find_extremals(&grid, &errors, n_extremals);
        
        // Step 4: Check convergence
        let max_change = extremals.iter()
            .zip(new_extremals.iter())
            .map(|(a, b)| (a - b).abs())
            .fold(0.0_f64, f64::max);
        
        extremals = new_extremals;
        
        if max_change < 1e-12 {
            break;
        }
    }
    
    // Recover filter coefficients
    let (a, _) = solve_alternation_system(
        &extremals, m, bands, desired, weights
    );
    recover_coefficients(&a, n_coeffs)
}
```

<ChallengesFailures>
  <ChallengeFailure type="bug" title="Barycentric Interpolation Instability">
    During step 1 of the exchange algorithm, we initially used a naive Lagrange interpolation polynomial. For high filter orders ($N > 128$), the Vandermonde matrix condition number exploded, causing catastrophic cancellation in floating-point math. The resulting filters had massive ripples instead of equiripple behavior. 
    **Fix**: Switched to the Barycentric form of Lagrange interpolation, which is numerically stable and faster ($O(N)$ evaluation).
  </ChallengeFailure>
  <ChallengeFailure type="bottleneck" title="JavaScript to WASM Bridge Overhead">
    Passing large arrays (bands, weights, desired) from JS to WASM on every render frame caused significant serialization overhead, neutralizing the performance gains of Rust.
    **Fix**: Pre-allocated memory buffers in WebAssembly memory and passed pointers from JS, zero-copy.
  </ChallengeFailure>
</ChallengesFailures>

<LessonsLearned>
  <Lesson title="WASM isn't a magic bullet for all performance issues">
    The overhead of sending data back and forth across the JS/WASM boundary can easily eclipse the raw compute gains if not careful. You must minimise memory crossings.
  </Lesson>
  <Lesson title="Numerical stability ruins algorithms on paper">
    Standard textbooks present algorithms in their pure mathematical form. In reality, IEEE 754 floating point requires rearranging equations (like Barycentric vs Standard Lagrange) to prevent catastrophic cancellation.
  </Lesson>
</LessonsLearned>
